## 1. 概述

### 1.1 编写目的
本手册旨在为 Q版方块人大乱斗项目提供标准化的开发实施指南，涵盖从环境搭建、开发流程、测试流程到部署上线的全过程，确保项目按计划高质量交付。

### 1.2 适用范围
本手册适用于 Q版方块人大乱斗（Blocky Fighter）的全部实施工作，包括 Unity 客户端开发、.NET8 帧同步服务器开发、美术资源制作与集成、测试及 Steam 发布。

### 1.3 术语定义
| 术语 | 定义 |
|------|------|
| 帧同步（Lockstep） | 网络同步方案，所有客户端运行相同逻辑，服务端仅转发输入帧 |
| 确定性（Deterministic） | 相同输入在任何客户端产生完全相同输出 |
| 连招系统 | 通过轻/重攻击组合触发不同招式的战斗系统 |
| 力量体系 | 武术/巫术/魔法/仙术四大角色分类，存在克制关系 |
| 双武器切换 | 战斗中可随时切换两把专属武器的机制 |
| Battle Pass（BP） | 赛季通行证，通过游戏活跃度推进度获取奖励 |

## 2. 实施总体策略

### 2.1 实施方法论
采用 **敏捷开发 + 迭代交付** 方法论：
- 以 2-3 周为一个迭代周期
- 每个迭代交付可运行的增量版本
- 持续集成、持续测试
- 快速响应变更

### 2.2 实施阶段划分
```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 准备阶段 │ → │ 开发阶段 │ → │ 测试阶段 │ → │ 打磨阶段 │ → │ 发布阶段 │ → │ 运营阶段 │
│ 环境搭建 │   │ 功能实现 │   │ 功能测试 │   │ 美术+音效│   │ Steam发布│   │ 赛季运营 │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

### 2.3 实施原则
1. **核心优先：** 战斗手感 > 画面表现 > 商业化功能
2. **快速验证：** 每个迭代产出可玩版本
3. **数据驱动：** 用数据（帧率/延迟/Bug率）衡量质量
4. **自动化优先：** 测试/构建/部署尽可能自动化
5. **文档同步：** 代码变更同步更新文档

# Unity 编辑器打开后等待编译完成，确认 Console 无错误

#### 3.4.2 服务端环境
```bash
# 步骤1：安装 .NET 8 SDK
# 从 https://dotnet.microsoft.com/download 下载安装

# 步骤2：验证安装
dotnet --version  # 应输出 8.x.x

# 步骤3：进入服务端目录
cd Q-BlockyFighter/Server

# 步骤4：还原依赖并编译
dotnet restore
dotnet build

# 步骤5：启动服务端
dotnet run
```

cd Q-BlockyFighter/Web

# 步骤3：安装依赖
npm install

# 步骤4：启动开发服务器
npm run dev

# 步骤5：浏览器打开 http://localhost:3000

#### 4.1.2 关键原则
- **帧同步逻辑与渲染严格分离：** 战斗逻辑不依赖 Unity 特定 API（如 Transform/Physics）
- **输入层抽象：** 客户端输入统一采集为帧数据，送入帧同步引擎
- **确定性保证：** 避免浮点数精度差异，使用定点数或整数运算

## 5. 开发规范

### 5.1 代码规范
- **语言：** C#（Unity + 服务端）
- **命名规范：**
  - 类名/方法名：PascalCase（如 `CombatSystem`, `PerformAttack`）
  - 变量名：camelCase（如 `currentHealth`, `attackDamage`）
  - 常量：UPPER_SNAKE_CASE（如 `MAX_HEALTH`, `FRAME_RATE`）
  - 私有字段：_camelCase（如 `_inputBuffer`）
- **代码注释：** 公共 API 必须有 XML 文档注释，核心逻辑行内注释
- **文件组织：** 按功能模块分目录（Combat/Character/Weapon/Network/UI/...）

### 5.2 分支策略
```
main          ← 稳定发布分支（对应 Steam 版本）
  └── dev     ← 开发主分支
       ├── feature/xxx  ← 功能分支
       ├── fix/xxx      ← Bug 修复分支
       └── art/xxx      ← 美术资源分支
```

### 5.3 Commit 规范
```
<type>(<scope>): <subject>

type: feat | fix | art | docs | refactor | test | chore
scope: combat | character | weapon | network | ui | server | ...
示例: feat(combat): 实现弹反机制
示例: fix(network): 修复帧同步丢帧问题
示例: art(character): 添加雾隐Q版模型
```

### 5.4 代码评审要求
| 项目 | 要求 |
|------|------|
| 评审覆盖率 | 核心模块（战斗/帧同步/网络）100% 双人审查 |
| 评审标准 | 功能正确性/代码规范/性能/安全性 |
| 自审要求 | 提交前必须自审，确保编译通过+基础测试 |

### 5.5 持续集成
| 阶段 | 工具 | 配置说明 |
|------|------|---------|
| 代码检查 | Unity 编译器 + IDE 静态分析 | 编译零错误零警告 |
| 自动构建 | Unity Build Pipeline | 每次打 Tag 自动构建 Windows exe |
| 版本管理 | Git Tag + 语义化版本 | v0.5.0, v0.6.0, v1.0.0 |

| 业务 | 在线人数/对局数 | 服务器日志 | 异常波动 | 每日报告 |

### 8.2 日志规范
| 日志级别 | 使用场景 | 示例 |
|---------|---------|------|
| ERROR | 服务器崩溃/帧同步异常 | 连接断开、帧数据异常 |
| WARN | 性能预警/异常状态 | 延迟过高、内存占用大 |
| INFO | 关键操作记录 | 玩家登录/登出、对局开始/结束 |
| DEBUG | 调试信息 | 帧数据、输入序列 |

### 8.3 常见问题处理（FAQ）
| 问题现象 | 可能原因 | 排查步骤 | 解决方案 |
|---------|---------|---------|---------|
| 无法连接服务器 | 端口未开放/防火墙 | 检查端口+防火墙规则 | 开放端口 |
| 游戏卡顿/掉帧 | 硬件不足/特效过多 | 查看 Unity Profiler | 降低画质设置 |
| 联机不同步 | 帧同步逻辑 bug | 检查确定性测试 | 修复逻辑 bug |
| 匹配超时 | 在线人数不足 | 检查匹配队列 | 调整匹配范围 |
