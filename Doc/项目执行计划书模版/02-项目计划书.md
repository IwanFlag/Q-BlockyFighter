## 1. 项目概述

### 1.1 项目背景

Q版方块人大乱斗是一款融合流星蝴蝶剑经典格斗玩法的多人实时动作竞技游戏。项目已完成Web端原型验证（V0.1-V0.3），核心战斗系统（轻攻击/重攻击/连招/闪避/格挡/弹反/体力系统）已通过浏览器端原型验证，帧同步联机对战已实现。现正在进行Unity引擎迁移（V0.4），计划以Unity 3D + .NET8帧同步服务器的技术栈完成正式版本开发，最终在Steam平台上线。

### 1.2 项目目标
| 编号 | 目标描述 | 衡量标准 | 优先级 |
|------|---------|---------|-------|
| G01 | 完成Unity引擎迁移，代码可编译运行 | V0.4版本发布，帧同步正常工作 | P0 |
| G02 | 实现全部12个角色的战斗系统 | 12角色可选择、技能可用、数据完整 | P0 |
| G03 | 实现1V1/5V5对战模式 | 可正常匹配、对局完成 | P0 |
| G04 | 实现训练场和闯关挑战模式 | 模式可进入、功能完整 | P1 |
| G05 | 完成商城/Battle Pass/排位系统 | 商业化闭环 | P1 |
| G06 | Steam上线，首月10万下载 | Steam页面上线、社区运营启动 | P1 |

### 1.3 项目范围
**包含（In Scope）：**
- 12个角色的完整战斗系统（含双武器切换、Q版/朋克版双风格）
- 1V1 BO5决斗模式 + 5V5团战模式
- 训练场模式（木人桩/武器架/Boss模拟/对战擂台）
- 闯关挑战模式（100波PVE）
- 排位系统（七段位）
- 商城系统（纯外观）
- Battle Pass赛季系统
- 帧同步服务器（.NET8）
- Steamworks集成

**不包含（Out of Scope）：**
- 移动端适配（后续版本）
- 手柄/平板支持（后续版本）
- 主机平台（PS/Xbox）（后续版本）
- VR模式

## 3. 技术方案

### 3.1 技术选型
| 层次 | 技术方案 | 选型理由 | 备选方案 |
|------|---------|---------|---------|
| 客户端引擎 | Unity 3D (2022.3 LTS) | 生态成熟、C#代码复用、跨平台好 | Godot、UE5 |
| 客户端语言 | C# | Unity原生语言，服务端也可复用 | — |
| 服务器 | .NET 8 | 高性能、C#生态统一 | Node.js、Go |
| 网络通信 | WebSocket | 帧同步方案成熟，浏览器兼容 | TCP/UDP |
| 帧同步 | 20fps逻辑帧 | 流星蝴蝶剑级别手感要求 | 60fps（带宽压力大） |
| 部署 | Windows Server / Linux | .NET8跨平台 | Docker |

### 3.3 核心模块设计
| 模块 | 职责 | 接口（输入/输出） | 依赖模块 |
|------|------|-----------------|---------|
| Core | 战斗核心逻辑：连招/体力/血量/伤害计算 | 输入指令→战斗状态 | — |
| Network | WebSocket通信、帧同步、消息序列化 | 网络帧→本地状态 | Core |
| Modes | 游戏模式管理（1V1/5V5/训练场/闯关） | 模式切换→对应逻辑 | Core, Map |
| AI | AI敌人行为（木人桩/野怪/Boss） | AI决策→输入指令 | Core |
| UI | HUD、菜单、商城、排位显示 | UI事件→游戏逻辑 | Core, Modes |
| Map | 地图加载、场景管理、碰撞检测 | 地图数据→场景渲染 | — |
| Utils | 工具类（数学/序列化/日志） | 通用接口 | — |

### 3.4 接口设计
| 接口名称 | 方向 | 协议 | 数据格式 | 调用方 | 频率 |
|---------|------|------|---------|-------|------|
| `/api/status` | 入 | HTTP | JSON | 客户端 | 按需 |
| `/api/rooms` | 入 | HTTP | JSON | 客户端 | 按需 |
| `join` | 入→出 | WebSocket | JSON | 客户端→服务器 | 按需 |
| `create_room` | 入→出 | WebSocket | JSON | 客户端→服务器 | 按需 |
| `input` | 入 | WebSocket | JSON | 客户端→服务器 | 20fps |
| `frame` | 出 | WebSocket | JSON | 服务器→客户端 | 20fps |
| `matchmake` | 入→出 | WebSocket | JSON | 客户端→服务器 | 按需 |

### 3.5 数据设计
主要数据模型：
- **Player**：playerId, name, characterId, skinId, rank, stats
- **Room**：roomId, players[], state, mode, mapId
- **Frame**：frameId, inputs[] (每个玩家的输入指令)
- **Character**：characterId, name, role, forceSystem, weapons[], skills[]
- **Match**：matchId, players[], result, duration, date

### 3.6 非功能性需求
| 维度 | 要求 | 验证方式 |
|------|------|---------|
| 性能 | 帧同步20fps，客户端渲染60fps，延迟<100ms | 压测+实机测试 |
| 可用性 | 服务器99%在线率 | 监控 |
| 安全性 | 服务端校验所有关键输入，客户端不信任原则 | 代码审查 |
| 可扩展性 | 服务器支持水平扩展，多房间独立运行 | 压测 |
| 可维护性 | 模块化设计，日志完整，方便调试 | 代码审查 |

### 6.2 高优先级风险应对预案

**[R01] 帧同步稳定性不足：**
- **触发条件：** 测试中发现同屏10人延迟>200ms或频繁掉线
- **应对步骤：** 1. 优化输入指令序列化格式 2. 增加服务端帧缓冲 3. 考虑客户端预测+回滚方案
- **所需资源：** 额外2-3周优化时间
- **升级条件：** 优化后仍无法满足10人同屏需求

**[R06] 个人精力有限：**
- **触发条件：** 任意里程碑延期超过2周
- **应对步骤：** 1. 重新评估优先级 2. 砍掉非核心功能 3. 外包部分工作
- **所需资源：** 外包预算¥10,000-30,000
- **升级条件：** 核心里程碑延期超过1个月
