## 1. 项目概述

### 1.1 项目背景
Q版方块人大乱斗是一款融合流星蝴蝶剑经典格斗、永劫无间美术风格、LOL UI 与角色设计的多人动作竞技游戏。项目采用「Web 原型验证 → Unity 正式开发」的渐进策略，先在浏览器中验证核心格斗手感与联机体验，再迁移至 Unity 引擎进行正式开发。

当前状态：Web 原型 V0.6 已完成，Unity 迁移 V0.4 代码已完成，处于 V0.5 全内容实现阶段。

### 1.2 项目目标
| 编号 | 目标描述 | 衡量标准 | 优先级 |
|------|---------|---------|-------|
| G01 | 完成 Unity 端全部 12 角色开发 | 12 角色均可在 Unity 端正常对战，连招/技能/双武器切换功能完整 | P0 |
| G02 | 实现 1V1 + 5V5 多人对战模式 | 帧同步联机对战稳定，10 人同屏 60fps | P0 |
| G03 | 完成训练场（木人桩/闯关/Boss 模拟/切磋） | 各功能区域可用，闯关排名系统正常 | P0 |
| G04 | 实现排位系统（七段位）+ 匹配机制 | 匹配时间 < 60s，段位计算准确 | P1 |
| G05 | 完成商业化系统（外观商城+Battle Pass） | 商城商品展示/购买/穿戴流程完整，BP 进度追踪正常 | P1 |
| G06 | 达到 Steam Early Access 发布标准 | 通过 Steam 审核，无 P0/P1 缺陷 | P0 |

## 3. 技术方案

### 3.1 技术选型
| 层次 | 技术方案 | 选型理由 | 备选方案 |
|------|---------|---------|---------|
| 游戏引擎 | Unity 3D（C#） | 生态成熟、跨平台好、C# 代码可复用 | UE5（画面顶级但学习曲线陡） |
| 网络通信 | WebSocket + 自定义协议 | 帧同步需要低延迟双向通信 | gRPC（更适合 RPC 场景） |
| 服务端 | .NET8 | 与 Unity C# 代码共享帧同步逻辑 | Node.js/Go（备选） |
| 帧同步引擎 | 确定性物理+输入同步 | 格斗游戏对公平性要求极高 | 状态同步（带宽消耗大） |
| 数据存储 | SQLite/JSON（初期） | 小团队快速迭代 | PostgreSQL（规模化后） |
| 部署 | Steam + 自建服务器 | Steam 提供分发+社区 | 其他平台（后续） |

### 3.3 核心模块设计
| 模块 | 职责 | 接口（输入/输出） | 依赖模块 |
|------|------|-----------------|---------|
| 战斗系统 | 格斗核心逻辑（攻击/连招/闪避/格挡/弹反/体力） | 输入：玩家操作 → 输出：伤害/状态变化 | 角色系统、武器系统 |
| 角色系统 | 12 角色属性/技能/力量体系 | 输入：角色选择 → 输出：属性/技能配置 | 武器系统 |
| 武器系统 | 双武器切换/连招表/品质系统 | 输入：武器操作 → 输出：连招/伤害 | 战斗系统 |
| 帧同步引擎 | 输入收集→帧推进→状态同步 | 输入：客户端操作帧 → 输出：权威游戏状态 | 网络层 |
| 匹配系统 | 1V1/5V5 匹配排队/段位计算 | 输入：匹配请求 → 输出：对局房间 | 排位系统 |
| 商城系统 | 外观商品展示/购买/穿戴 | 输入：购买请求 → 输出：商品发放 | 数据存储 |
| BP 系统 | 赛季通行证进度/奖励 | 输入：游戏行为 → 输出：进度/奖励 | 数据存储 |

### 3.4 接口设计
| 接口名称 | 方向 | 协议 | 数据格式 | 调用方 | 频率/QPS |
|---------|------|------|---------|-------|---------|
| 玩家输入帧 | 客户端→服务器 | WebSocket | Binary | 战斗系统 | 60次/秒 |
| 游戏状态帧 | 服务器→客户端 | WebSocket | Binary | 所有客户端 | 60次/秒 |
| 匹配请求 | 客户端→服务器 | WebSocket | JSON | 匹配系统 | 按需 |
| 商城查询 | 客户端→服务器 | HTTP/WS | JSON | 商城系统 | 按需 |
| 排名查询 | 客户端→服务器 | HTTP/WS | JSON | 排位系统 | 按需 |

### 3.5 数据设计
- **玩家数据**：账号信息、角色解锁状态、皮肤/外观拥有情况、段位/积分、BP 进度
- **对局数据**：对局回放（输入帧序列）、战绩统计、MVP 数据
- **商城数据**：商品目录、价格、库存、购买记录
- **排名数据**：段位分数、全球/好友排行榜

### 3.6 非功能性需求
| 维度 | 要求 | 验证方式 |
|------|------|---------|
| 性能 | 10 人同屏 60fps，输入延迟 < 50ms | 实机测试+压测 |
| 可用性 | 服务器 99% 可用性 | 监控 |
| 安全性 | 服务端权威校验，防外挂/篡改 | 代码审查+安全测试 |
| 可扩展性 | 支持新增角色/地图/模式 | 模块化设计 |
| 可维护性 | 代码注释率 > 30%，日志完整 | 代码审查 |

### 6.2 高优先级风险应对预案

**[R01] 帧同步弱网体验差：**
- **触发条件：** 玩家网络延迟 > 150ms 或丢包率 > 5%
- **应对步骤：** 1. 启用客户端预测+权威回滚 2. 动态调整帧率（降低至 30fps） 3. 提示玩家切换网络
- **所需资源：** 额外 1-2 周开发时间
- **升级条件：** 大量玩家反馈延迟问题

**[R03] 角色平衡性问题：**
- **触发条件：** 某角色胜率偏离 50% 超过 ±10%
- **应对步骤：** 1. 数据分析定位问题 2. 数值微调（伤害/冷却/体力消耗） 3. 发布热更新
- **所需资源：** 数据埋点+每周数据分析
- **升级条件：** 某角色被玩家广泛投诉为 OP/废物

**[R05] 开发周期超出预期：**
- **触发条件：** 里程碑延期超过 2 周
- **应对步骤：** 1. 评估可削减功能 2. 优先保证核心战斗体验 3. 非核心功能移至 V1.1+
- **可削减范围：** 部分地图、部分角色朋克版皮肤、反作弊系统
- **升级条件：** 延期超过 1 个月
